### Post-build audit pass (04:00–05:00)

Reviewed all newly-built files for security, robustness, and correctness. Findings + fixes:

**Fixed: `lib/ai/twin-prompts.ts` callTwinAI lacks fetch timeout.**
- Symptom: A hung OpenRouter call would block API route indefinitely; serverless function would time out at platform level (300s) with no graceful error.
- Fix: AbortController with 20s default timeout (configurable via `opts.timeoutMs`). Returns structured `"AI request timed out"` error on abort. Also added `HTTP-Referer` + `X-Title` headers (OpenRouter analytics) and propagated `data.usage` in the success return so callers can log token counts.

**Fixed: payment-match + reply-draft don't surface AI usage.**
- Symptom: Phase 0 cost measurement is script-only; production calls leave no token count footprint, defeating "measure under load" intent.
- Fix: Both endpoints now return `usage: result.usage ?? null` in the JSON response. The admin AI test console naturally surfaces this to Aldi during smoke test.

**Fixed: Phase 0 export auto-decision logic stuck in "PARTIAL".**
- Symptom: Triggers #1 (AI cost), #4 (Ariff), #5 (SSM) hard-coded as PENDING because they're tracked outside the dashboard. `allPass` evaluation included them, so the recommendation could never reach "PASS" — even with all in-dashboard signals unanimous.
- Fix: Added `external?: boolean` flag to `TriggerEval`, marked the 3 external triggers, and excluded them from `allPass`/`allPending`/`killTriggered` computation. The decision text now explicitly tells Aldi to verify the 3 externals manually before locking Phase 1.

**Fixed: Phase 0 dashboard header miscounted triggers.**
- Symptom: Header read "6 kill + 1 rebrand-flag" but the table shows 12 sub-metrics (the API breaks each SYNTHESIS-level trigger into multiple measurable rows). Aldi reading dashboard would think a synthesis-document conflict.
- Fix: Header now reads "12 in-dashboard sub-metrics aggregating to 4 of the 7 SYNTHESIS-level triggers" + caveat about external 3.

**Fixed: OrderForm double-decodes reorder items param.**
- Symptom: `decodeURIComponent` was called on `searchParams.get("items")`, but `URLSearchParams.get()` already URI-decodes. Bare `%` literals in product names (e.g. "10% off product") trigger `URIError: URI malformed` on the second decode, which the try/catch swallows — leaving the cart empty without telling the merchant the prefill failed.
- Fix: Drop the redundant `decodeURIComponent`, parse the already-decoded string directly. Inline comment explains the trap so the next person doesn't re-add it.

**Fixed: Phase 0 routes accept dead `"founder"` role string.**
- Symptom: All 5 phase-0 admin routes had `role !== "admin" && role !== "founder"`, but `"founder"` isn't in `USER_ROLES` and no migration sets it. The check was always equivalent to `role !== "admin"` — confusing future maintainers and inviting a misconfigured role to "fix" the symptom rather than the cause.
- Fix: Removed the dead `"founder"` clause from all 5 routes (main, export, interviews, distribution, smoke-test). Aldi's role on the profile must be exactly `"admin"`.

**Fixed: Restraint violation in `copy.empathy.midRush`.**
- Symptom: The mid-rush empathy moment said "I'll handle customer chat. You focus on cooking." Per bible v1.2 + 10-item refusal list, Tokoflow NEVER auto-replies — Foreground Assist drafts, merchant sends. The verb "handle" implies takeover.
- Fix: Rephrased to "I'll draft replies so you can keep cooking. Tap to send." Same warmth, accurate restraint.

**Fixed: Phase 0 entry forms swallow network errors.**
- Symptom: All 3 entry forms (interviews, distribution, smoke-test) had `try/finally` with no `catch`. If `fetch` itself threw (offline, CORS, DNS), `setSaving(false)` reset the spinner but no toast told the admin what went wrong — just silent failure with no visible feedback beyond the form re-enabling.
- Fix: Added explicit `catch (err)` blocks emitting toast with error message. Now offline/connection errors during Phase 0 logging produce visible feedback so Aldi knows to retry rather than wonder if the click registered.

**Hardened: PhotoMagicEntry response-shape validation.**
- Symptom: After a successful 200 response, the code did `data.products.length === 0` without first verifying `data.products` was an array. A malformed/legacy response would throw `TypeError: Cannot read properties of undefined (reading 'length')`, caught by the outer try/catch and showing the generic "Couldn't process photo" toast — masking the actual contract violation.
- Fix: Added explicit shape guard — check `data` exists, `Array.isArray(data.products)`, then length. Falls through to the existing low-confidence path with a specific toast pointing to "try another photo or manual setup."

**Fixed: Production/script prompt divergence (REPLY_DRAFT_PROMPT).**
- Symptom: `lib/ai/twin-prompts.ts` REPLY_DRAFT_PROMPT differed from `scripts/phase-0/ai-cost/prompts/foreground-assist.ts` REPLY_DRAFT_PROMPT in both wording and response shape (production returns `tone`/`alternative`; script expected `needs_merchant_attention`/`alternative_drafts`). CLAUDE.md says "Keep these in sync" — they weren't. AI cost measurement was using a longer, voice-example-heavy prompt while production used a tighter restraint-focused one, meaning measured cost overestimated production reality.
- Fix: Synced the script's REPLY_DRAFT_PROMPT to match production verbatim. Added a header comment marking production as source of truth (since the deployed `/api/assist/reply-draft` route depends on the response shape). Cost measurement will now accurately reflect what production deploys.
- Out-of-scope (intentional): `COMPLAINT_DRAFT_PROMPT` and `PRICING_WHISPER_PROMPT` remain in the script but absent from production. They measure cost forecast for Year-2 features. When those features ship to production, port the script versions over (they're already synthesis-aligned).
