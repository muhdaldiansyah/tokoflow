# Mobile UX Overhaul — 19 Maret 2026

> Audit lengkap semua masalah yang ditemukan dan diperbaiki di Tokoflow Mobile App

**Tanggal:** 19 Maret 2026
**Status:** 32 issues ditemukan, 30 fixed, 2 open

---

## A. Tempel Chat & AI Parsing (7 issues)

### A1. Tempel Chat langsung ke Detail Pesanan
**Masalah:** Setelah Tempel Chat di home screen, pesanan langsung dibuat dan user diarahkan ke halaman detail — tanpa kesempatan mengisi pelanggan, tanggal kirim, atau status bayar.
**Penyebab:** `handleAIParsedItems()` di home screen langsung memanggil `createOrder.mutate()`.
**Fix:** Navigasi ke `/orders/new?prefillOrder=<JSON>` — user bisa review dan edit sebelum submit.
**Status:** FIXED

### A2. AI hanya parse item, tidak parse metadata
**Masalah:** `parseTextToItems()` hanya return `OrderItem[]` (nama, qty, harga). Tidak ada customer name, tanggal, status bayar, diskon, catatan.
**Penyebab:** Endpoint salah (`/api/recap/analyze` yang tidak handle `parse_order`) dan tidak ada logic metadata.
**Fix:** Ganti ke `/api/voice/parse` yang sudah extract semua metadata. Tambah `fallbackParseOrder()` untuk local parsing.
**Status:** FIXED

### A3. Image parsing endpoint salah
**Masalah:** `parseImageToItems()` call `/api/recap/analyze` dengan `type: "parse_order_image"` — endpoint ini tidak exist.
**Penyebab:** Copy-paste dari text parsing, endpoint tidak diupdate.
**Fix:** Ganti ke `/api/image/parse` dengan format payload yang benar (`data:image/jpeg;base64,...`).
**Status:** FIXED

### A4. Backend image parse tidak extract metadata
**Masalah:** `/api/image/parse` di web backend hanya return `{ items }`, tidak extract customer/tanggal/bayar.
**Penyebab:** Prompt Gemini hanya minta items.
**Fix:** Update prompt untuk extract semua metadata (sama seperti `/api/voice/parse`). Response sekarang return full `ParsedOrder`.
**Status:** FIXED

### A5. Parsed items tidak dicocokkan dengan katalog produk
**Masalah:** AI bisa return item yang tidak ada di katalog toko — harga 0, nama tidak sesuai.
**Penyebab:** Tidak ada post-processing setelah AI parse.
**Fix:** Tambah `matchItemsToProducts()` — exact match dulu, lalu fuzzy match. Item yang tidak cocok dibuang.
**Status:** FIXED

### A6. Fallback parser return item non-katalog
**Masalah:** `fallbackParse()` menambah item dengan `price: 0` jika tidak cocok di katalog.
**Penyebab:** Ada `else if` yang push item tanpa match.
**Fix:** Hapus `else if` — hanya item yang cocok katalog yang ditambahkan.
**Status:** FIXED

### A7. Customer name tidak terparse di fallback
**Masalah:** Input "kue coklat 3, rano karno tanggal 24 mar, lunas" → nama "rano karno" tidak terdeteksi.
**Penyebab:** Fallback regex terlalu ketat (hanya "nama:", "an.", dll). "rano karno tanggal 24 mar" dianggap satu segment.
**Fix:** 3 layer detection: (1) keyword explicit, (2) prefix bu/pak/mbak, (3) smart guess — strip tanggal/bayar phrases, sisa text = nama.
**Status:** FIXED

## G. Database & API (5 issues)

### G1. Customer stats tidak otomatis sync
**Masalah:** `total_orders`, `total_spent`, `last_order_at` di customers table tidak update saat order di-edit/hapus/bayar.
**Penyebab:** Stats hanya di-recalculate saat order CREATE, tidak saat UPDATE atau DELETE.
**Fix:** Database trigger `trg_recalculate_customer_stats` — auto-recalculate pada setiap INSERT/UPDATE/DELETE di orders table.
**Status:** FIXED (migration 053)

### G2. Duplicate customers (case mismatch)
**Masalah:** "zorro" dan "Zorro" jadi 2 customer berbeda. "luffy" ada 2 record (phone berbeda).
**Penyebab:** Customer matching hanya by phone exact match. Tidak ada case-insensitive name matching.
**Fix:** (1) Order creation sekarang match by phone first, then by name case-insensitive. (2) Cleanup migration merge duplicates. (3) Name normalized saat matching.
**Status:** FIXED (migration 054 + order API update)

### G3. Orphan orders (no customer linked)
**Masalah:** 7+ orders dengan `customer_name` tapi `customer_id = null`.
**Penyebab:** Jika tidak ada phone, customer tidak dicreate/dilink.
**Fix:** Order creation sekarang juga match by name jika tidak ada phone. Cleanup migration link orphan orders ke existing customers atau create baru.
**Status:** FIXED

### G4. `/api/recap/analyze` pakai cookie-only auth
**Masalah:** Mobile app kirim Bearer token → 401 Unauthorized.
**Penyebab:** Endpoint pakai `createClient()` (cookies) bukan `getAuthenticatedClient(req)` (Bearer + cookies).
**Fix:** Ganti ke `getAuthenticatedClient(request)` di GET dan POST handler.
**Status:** FIXED

### G5. `/api/orders/summary` timezone dan unpaid count salah
**Masalah:** "Belum Bayar" count 15 padahal seharusnya 16. "Today" pakai UTC bukan WIB.
**Penyebab:** (1) `unpaidCount` query exclude status "done" dan pakai `paid_amount = 0`. (2) `startOfDay`/`endOfDay` pakai UTC `new Date().toISOString()`.
**Fix:** (1) Ganti ke `payment_status IN ('unpaid', 'partial')` tanpa status filter. (2) Pakai WIB timezone untuk "today".
**Status:** FIXED

## H. Masalah yang Masih OPEN

### H1. Monthly card di home screen menampilkan 30 pesanan, bukan 31
**Masalah:** Database punya 31 orders, tapi monthly recap API return 30. Service role query return 31, tapi user JWT query mungkin return 30.
**Kemungkinan penyebab:** RLS policy issue pada 1 order yang dibuat via public order link atau service role context.
**Status:** OPEN — resolved by clean slate (delete all test data). Perlu monitor di production data.
**Severity:** Low — hanya terjadi pada legacy test data.

### H2. Date parsing edge cases di fallback parser
**Masalah:** "tanggal 1" di bulan Februari bisa resolve ke 1 Februari (bulan ini) atau 1 Maret (bulan depan). Logic saat ini: jika bulan < current month → year + 1.
**Status:** OPEN — acceptable for now, AI handles most cases correctly.
**Severity:** Low
